## 1.1 什么是编译器

### 1.1.1 定义

**编译器（Compiler）** 是一种将源代码（高级编程语言）翻译为目标代码（通常是机器码或另一种低级表示）的程序。这个翻译过程称为**编译（Compilation）**。

从形式化角度看，编译器是一个从源语言到目标语言的映射函数：

```
Compiler: SourceLanguage → TargetLanguage
```

但这个映射不是简单的逐行翻译——编译器需要理解源代码的**结构和语义**，然后生成等价的、高效的目标代码。

### 1.1.2 历史简述

| 年代 | 里程碑 | 意义 |
|------|--------|------|
| 1952 | Grace Hopper 开发 A-0 系统 | 第一个编译器的雏形 |
| 1957 | IBM 的 FORTRAN 编译器 | 第一个成熟的优化编译器，历时18人年开发 |
| 1970s | Unix 的 `cc` 和 `pcc` | 可移植 C 编译器，影响了后续所有 C 编译器 |
| 1987 | GCC 1.0 发布 | GNU 编译器集合，成为开源编译器的基石 |
| 2000s | LLVM 项目启动 | 模块化编译器基础设施，革新了编译器架构 |
| 2010s+ | JIT 编译器普及 | V8、LuaJIT 等将编译技术带入运行时 |

### 1.1.3 GCC 将 C 编译为可执行文件的过程

以 GCC 编译一个 C 程序为例，完整过程如下：

```bash
# 一步到位：源代码 → 可执行文件
gcc example.c -o example

# 等价于以下四个阶段的组合：
gcc -E example.c -o example.i    # 1. 预处理（Preprocessing）
gcc -S example.i -o example.s    # 2. 编译（Compilation）：C → 汇编
gcc -c example.s -o example.o    # 3. 汇编（Assembly）：汇编 → 机器码（目标文件）
gcc example.o -o example          # 4. 链接（Linking）：目标文件 → 可执行文件
```

**各阶段详解：**

**阶段 1：预处理（Preprocessing）**
- 处理 `#include`、`#define`、`#ifdef` 等预处理指令
- 展开宏，插入头文件内容
- 删除注释
- 输出 `.i` 文件（纯 C 代码，无预处理指令）

**阶段 2：编译（Compilation）**
- 这是编译原理课程的核心关注点
- 将预处理后的 C 代码翻译为汇编代码
- 内部经历：词法分析 → 语法分析 → 语义分析 → 中间代码生成 → 优化 → 目标代码生成
- 输出 `.s` 文件（汇编代码）

**阶段 3：汇编（Assembly）**
- 将汇编代码翻译为机器指令
- 生成目标文件（`.o` / `.obj`），格式为 ELF（Linux）、PE/COFF（Windows）、Mach-O（macOS）
- 目标文件包含机器码、数据段、符号表、重定位信息

**阶段 4：链接（Linking）**
- 将多个目标文件和库合并为一个可执行文件
- 解析符号引用（函数调用、全局变量）
- 处理静态库（`.a` / `.lib`）和动态库（`.so` / `.dll`）
- 分配最终的虚拟地址空间

```
┌─────────────────────────────────────────────────────────────┐
│                    GCC 编译完整流程                          │
│                                                             │
│  example.c                                                  │
│      │                                                      │
│      ▼                                                      │
│  ┌──────────┐    example.i    ┌──────────┐    example.s     │
│  │ 预处理器  │ ──────────────→ │  编译器   │ ─────────────→  │
│  │ (cpp)    │                 │  (cc1)   │                  │
│  └──────────┘                 └──────────┘                  │
│                                   │                         │
│                                   ▼                         │
│                             example.s                       │
│                                   │                         │
│                                   ▼                         │
│                              ┌──────────┐    example.o      │
│                              │  汇编器   │ ─────────────→   │
│                              │  (as)    │                   │
│                              └──────────┘                   │
│                                   │                         │
│                                   ▼                         │
│                              example.o                       │
│                                   │                         │
│                                   ▼                         │
│                              ┌──────────┐    example        │
│                              │  链接器   │ ─────────────→   │
│                              │  (ld)    │                   │
│                              └──────────┘                   │
│                                   ↑                         │
│                              libc.a, crt1.o ...             │
│                          (C 运行时库和启动代码)              │
└─────────────────────────────────────────────────────────────┘
```

### 1.1.4 MinGW 说明

**MinGW（Minimalist GNU for Windows）** 是 GCC 在 Windows 平台上的移植版本。

| 特性 | 说明 |
|------|------|
| **本质** | 一套 Windows 原生的 GCC 工具链 |
| **输出格式** | 生成 Windows PE 格式的 `.exe` 和 `.dll` |
| **C 运行时** | 使用 Microsoft 的 `msvcrt.dll`（微软 C 运行时） |
| **与 Cygwin 的区别** | MinGW 生成原生 Windows 程序，不依赖 POSIX 模拟层 |
| **MSYS2** | MinGW 的现代发行版，提供包管理器 `pacman`，方便安装工具链 |

```bash
# MinGW 编译过程与 Linux 上的 GCC 完全一致
gcc example.c -o example.exe

# 同样支持分步编译
gcc -E example.c -o example.i
gcc -S example.i -o example.s
gcc -c example.s -o example.o
gcc example.o -o example.exe
```

> **实践提示**：在 Windows 上，推荐使用 MSYS2 安装 MinGW-w64 工具链，它同时支持 32 位和 64 位目标，且工具链版本更新及时。

## 1.2 什么是解释器

### 1.2.1 定义

**解释器（Interpreter）** 是一种直接执行源代码（或某种中间表示）的程序。与编译器不同，解释器不生成独立的目标文件——它在运行时逐条读取、分析并执行指令。

从形式化角度看，解释器是一个执行函数：

```
Interpreter: Program × Input → Output
```

解释器本身就是目标机器——它"理解"源语言的语义，并直接产生执行效果。

### 1.2.2 CPython 的工作方式

CPython（Python 的官方实现）采用**编译+虚拟机**的混合架构：

```
┌───────────────────────────────────────────────────────────┐
│                   CPython 执行流程                         │
│                                                           │
│  source.py                                                │
│      │                                                    │
│      ▼                                                    │
│  ┌─────────────────────┐                                  │
│  │    编译阶段 (C实现)   │                                  │
│  │                     │                                  │
│  │  词法分析 → Token流  │                                  │
│  │       ↓             │                                  │
│  │  语法分析 → AST      │                                  │
│  │       ↓             │                                  │
│  │  编译为字节码        │                                  │
│  └─────────┬───────────┘                                  │
│            ▼                                              │
│  ┌─────────────────────┐                                  │
│  │  .pyc 字节码文件     │   (可选：缓存到 __pycache__/)     │
│  │  (平台无关的二进制)   │                                  │
│  └─────────┬───────────┘                                  │
│            ▼                                              │
│  ┌─────────────────────┐                                  │
│  │   Python 虚拟机      │                                  │
│  │   (CEval 循环)       │                                  │
│  │                     │                                  │
│  │   while True:       │                                  │
│  │     opcode = next() │                                  │
│  │     dispatch(opcode)│  ← 基于栈的虚拟机                 │
│  └─────────────────────┘                                  │
└───────────────────────────────────────────────────────────┘
```

**关键概念：**

- **源代码 → AST**：CPython 的编译器（`Python/compile.c`）将源码解析为抽象语法树
- **AST → 字节码**：AST 被编译为字节码指令序列（`.pyc` 文件）
- **虚拟机执行**：`Python/ceval.c` 中的 `eval_frame` 函数实现了一个巨大的 `switch` 语句，逐条执行字节码

```python
# 查看 Python 字节码的示例
import dis

def add(a, b):
    return a + b

dis.dis(add)
# 输出类似：
#   2           0 LOAD_FAST                0 (a)
#               2 LOAD_FAST                1 (b)
#               4 BINARY_ADD
#               6 RETURN_VALUE
```

> **核心洞察**：CPython 不是纯粹的解释器——它先**编译**为字节码，再**解释**执行字节码。这种两阶段架构在性能和实现复杂度之间取得了很好的平衡。

### 1.2.3 V8 引擎的混合 JIT 架构

Google 的 V8 引擎（Chrome 和 Node.js 的 JavaScript 引擎）代表了现代语言运行时的典型设计：

```
┌───────────────────────────────────────────────────────────┐
│                      V8 引擎架构                           │
│                                                           │
│  JavaScript 源代码                                         │
│      │                                                    │
│      ▼                                                    │
│  ┌───────────┐                                            │
│  │  Parser   │  → AST                                     │
│  └─────┬─────┘                                            │
│        ▼                                                  │
│  ┌───────────────────────────────────────────────┐        │
│  │              Ignition (解释器)                  │        │
│  │                                               │        │
│  │  AST → 字节码 → 逐条解释执行                    │        │
│  │  (启动速度快，但执行较慢)                        │        │
│  │                                               │        │
│  │  同时收集类型反馈（Type Feedback）               │        │
│  └───────────────────┬───────────────────────────┘        │
│                      │                                    │
│         热点函数被多次执行？                               │
│                      │ 是                                 │
│                      ▼                                    │
│  ┌───────────────────────────────────────────────┐        │
│  │           TurboFan (优化编译器)                 │        │
│  │                                               │        │
│  │  字节码 + 类型反馈 → 优化的机器码               │        │
│  │                                               │        │
│  │  优化策略：                                    │        │
│  │  • 内联缓存（Inline Cache）                    │        │
│  │  • 逃逸分析（Escape Analysis）                 │        │
│  │  • 类型特化（Type Specialization）             │        │
│  │  • 死代码消除                                  │        │
│  │                                               │        │
│  │  如果类型假设失败 → 去优化（Deoptimization）     │        │
│  │  回退到 Ignition 重新解释执行                    │        │
│  └───────────────────────────────────────────────┘        │
└───────────────────────────────────────────────────────────┘
```

**V8 的分层策略：**

| 层级 | 组件 | 角色 | 特点 |
|------|------|------|------|
| 第 0 层 | Parser | 源码 → AST | 快速解析 |
| 第 1 层 | Ignition | AST → 字节码 → 执行 | 启动快，内存占用低 |
| 第 2 层 | TurboFan | 热点代码 → 优化机器码 | 执行极快，编译较慢 |

> **为什么需要分层？** 编译优化需要时间。如果所有代码都用 TurboFan 编译，程序启动会很慢。Ignition 让程序能立即开始执行，只有真正"热"的代码才会被深度优化。

### 1.2.4 LuaJIT 的 JIT 策略

LuaJIT 是 Lua 语言的高性能实现，由 Mike Pall 开发，以其极高的性能著称。它的 JIT 策略与 V8 有显著不同：

```
┌───────────────────────────────────────────────────────────┐
│                   LuaJIT 架构                              │
│                                                           │
│  Lua 源代码                                                │
│      │                                                    │
│      ▼                                                    │
│  ┌─────────────────────┐                                  │
│  │   字节码编译器        │  → LuaJIT 字节码                 │
│  │   (前端，用 C 实现)   │                                  │
│  └─────────┬───────────┘                                  │
│            ▼                                              │
│  ┌─────────────────────────────────────────────┐          │
│  │         LuaJIT 虚拟机                        │          │
│  │                                             │          │
│  │  执行方式：手写汇编实现的快速解释器            │          │
│  │  (比 C 实现的 switch 循环快很多)              │          │
│  │                                             │          │
│  │  同时追踪：                                  │          │
│  │  • 热点计数器（Hot Count）                    │          │
│  │  • 字节码执行路径                             │          │
│  └─────────────────┬───────────────────────────┘          │
│                    │                                      │
│       热点回边（Hot Loop）触发？                           │
│                    │ 是                                   │
│                    ▼                                      │
│  ┌─────────────────────────────────────────────┐          │
│  │         Trace Compiler（踪迹编译器）          │          │
│  │                                             │          │
│  │  策略：不是编译整个函数，而是编译             │          │
│  │  "执行踪迹"（Execution Trace）               │          │
│  │                                             │          │
│  │  1. 记录一条从循环入口到出口的执行路径        │          │
│  │  2. 将踪迹编译为线性的机器码                  │          │
│  │  3. 通过"侧出口"（Side Exit）处理            │          │
│  │     类型变化和分支跳转                        │          │
│  │                                             │          │
│  │  优势：                                      │          │
│  │  • 编译速度极快（踪迹是线性的）               │          │
│  │  • 可以跨越函数边界（跨函数内联）             │          │
│  │  • 代码膨胀小（只编译热路径）                 │          │
│  └─────────────────────────────────────────────┘          │
└───────────────────────────────────────────────────────────┘
```

**LuaJIT 的核心创新：**

| 特性 | 说明 |
|------|------|
| **Trace-based JIT** | 编译执行踪迹而非整个函数，编译开销极低 |
| **Side Exit** | 当类型假设失败时，从优化代码"侧出口"跳回解释器 |
| **跨函数内联** | 追踪可以跨越函数调用边界，实现深度内联 |
| **FFI 编译器** | 对 C 函数调用也进行 JIT 编译，消除 FFI 开销 |
| **SSA IR** | 内部使用静态单赋值（SSA）中间表示进行优化 |

> **LuaJIT vs V8 的关键区别**：V8 采用方法级编译（Method-based），LuaJIT 采用踪迹级编译（Trace-based）。LuaJIT 的方式在循环密集型代码上尤其高效，因为它只编译实际执行的路径。

## 1.3 编译器 vs 解释器

### 1.3.1 执行方式对比

```
┌────────────── 编译器 ──────────────┐    ┌────────────── 解释器 ──────────────┐
│                                    │    │                                    │
│  源代码 ──编译──→ 可执行文件        │    │  源代码 ──读取──→ 逐条执行          │
│                    ↓               │    │                    ↓               │
│              直接在CPU上运行        │    │              在解释器内运行         │
│                                    │    │                                    │
│  [编译阶段]  ████████  →  程序      │    │  [无编译阶段]                       │
│  [运行阶段]  ██████████████████    │    │  [运行阶段]  ██████████████████████ │
│                                    │    │                                    │
└────────────────────────────────────┘    └────────────────────────────────────┘
```

### 1.3.2 全面对比

| 维度 | 编译器 | 解释器 |
|------|--------|--------|
| **执行方式** | 先编译为机器码，再直接执行 | 逐条读取并执行源代码/字节码 |
| **运行时性能** | 高（机器码直接执行，可深度优化） | 较低（每条指令都需要解释开销） |
| **启动时间** | 慢（需要先完成编译） | 快（直接开始执行） |
| **开发效率** | 较低（编译-链接周期） | 高（修改后立即运行） |
| **错误发现** | 编译时发现类型错误、语法错误 | 运行时才发现错误 |
| **可移植性** | 需要为每个平台编译 | 源代码/字节码跨平台（只要有解释器） |
| **调试** | 需要符号信息，优化后可能困难 | 更容易映射回源代码 |
| **内存占用** | 编译时内存大，运行时紧凑 | 运行时需要解释器本身 |
| **典型代表** | C, C++, Rust, Go | Python, Ruby, 早期 JavaScript |

### 1.3.3 混合模式——现代语言的趋势

现代语言实现越来越倾向于**混合模式**，结合编译和解释的优势：

```
┌─────────────────────────────────────────────────────────────┐
│                    语言实现光谱                               │
│                                                             │
│  纯编译                                              纯解释  │
│  ←─────────────────────────────────────────────────→        │
│                                                             │
│  C    C++    Rust    Go    Java    C#    JS(V8)    Lua    Python │
│  │     │      │      │      │      │      │       │        │  │
│  │     │      │      │      │      │      │       │        │  │
│  AOT   AOT    AOT    AOT    JIT    JIT   JIT    JIT    解释+编译│
│  编译  编译   编译   编译   +VM    +VM   +解释   +解释     +JIT │
│                                                             │
│  AOT = Ahead-of-Time（提前编译）                             │
│  JIT = Just-in-Time（即时编译）                              │
└─────────────────────────────────────────────────────────────┘
```

**典型的混合模式实现：**

| 语言/运行时 | 策略 | 说明 |
|------------|------|------|
| **Java (HotSpot)** | 解释 + 分层 JIT | 先解释执行，C1 编译器处理温代码，C2 编译器处理热代码 |
| **C# (.NET)** | 先 AOT（RyuJIT），后动态优化 | .NET 6+ 支持 NativeAOT 完全提前编译 |
| **JavaScript (V8)** | 解释器 (Ignition) + JIT (TurboFan) | 分层编译，带去优化回退 |
| **Python 3.13+** | 解释器 + 自适应特化 | 3.13 引入 free-threaded 和 JIT 实验 |
| **LuaJIT** | 快速解释器 + Trace JIT | 极致的分层优化 |
| **Julia** | LLVM JIT | 首次调用时编译，之后直接执行机器码 |

## 1.4 编译器的主要阶段

编译器的内部工作可以分解为一系列有序的阶段。每个阶段完成一个特定的转换任务，将程序从一种表示逐步变换为另一种表示。

### 1.4.0 贯穿示例代码

以下 C 代码将贯穿本节所有阶段的演示：

```c
// example.c — 贯穿示例
int square(int n) {
    return n * n;
}

int sum_of_squares(int n) {
    int total = 0;
    for (int i = 1; i <= n; i++) {
        total = total + square(i);
    }
    return total;
}

int main() {
    int result = sum_of_squares(5);
    return result;
}
```

### 1.4.1 完整流程总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                     编译器阶段完整流程                                │
│                                                                     │
│  ┌──────────┐                                                       │
│  │  源代码   │  "int square(int n) { return n * n; }"               │
│  └────┬─────┘                                                       │
│       ▼                                                             │
│  ┌──────────────────┐                                               │
│  │    词法分析        │  (Lexical Analysis / Scanning)                │
│  │    (Lexer)        │                                               │
│  └────┬─────────────┘                                               │
│       ▼                                                             │
│  ┌──────────┐                                                       │
│  │  Token流  │  [INT, ID("square"), LPAREN, INT, ID("n"), RPAREN,  │
│  │           │   LBRACE, RETURN, ID("n"), STAR, ID("n"), SEMI,     │
│  │           │   RBRACE]                                           │
│  └────┬─────┘                                                       │
│       ▼                                                             │
│  ┌──────────────────┐                                               │
│  │    语法分析        │  (Syntax Analysis / Parsing)                  │
│  │    (Parser)       │                                               │
│  └────┬─────────────┘                                               │
│       ▼                                                             │
│  ┌──────────┐                                                       │
│  │   AST    │  (抽象语法树)                                          │
│  └────┬─────┘                                                       │
│       ▼                                                             │
│  ┌──────────────────┐                                               │
│  │    语义分析        │  (Semantic Analysis)                          │
│  │                   │  类型检查、作用域解析、符号表填充               │
│  └────┬─────────────┘                                               │
│       ▼                                                             │
│  ┌──────────┐                                                       │
│  │   IR     │  (中间代码表示)                                        │
│  └────┬─────┘                                                       │
│       ▼                                                             │
│  ┌──────────────────┐                                               │
│  │    优化            │  (Optimization)                               │
│  │                   │  常量折叠、死代码消除、循环优化等              │
│  └────┬─────────────┘                                               │
│       ▼                                                             │
│  ┌──────────────┐                                                   │
│  │   目标代码    │  汇编代码 / 机器码 / 字节码                       │
│  └──────────────┘                                                   │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.4.2 阶段一：词法分析（Lexical Analysis）

**任务**：将源代码字符流分割为有意义的词法单元（Token）。

词法分析器（Lexer / Scanner）逐字符读取源代码，根据预定义的模式（通常用正则表达式描述）将其分组为 Token。

```
输入（字符流）:
  i n t   s q u a r e ( i n t   n )   {   r e t u r n   n   *   n ; }

输出（Token流）:
  ┌──────────┬────────────┬──────────────────┐
  │ Token类型 │  Token值    │  位置 (行:列)     │
  ├──────────┼────────────┼──────────────────┤
  │ INT      │  "int"     │  1:1             │
  │ ID       │  "square"  │  1:5             │
  │ LPAREN   │  "("       │  1:12            │
  │ INT      │  "int"     │  1:13            │
  │ ID       │  "n"       │  1:17            │
  │ RPAREN   │  ")"       │  1:18            │
  │ LBRACE   │  "{"       │  1:20            │
  │ RETURN   │  "return"  │  2:5             │
  │ ID       │  "n"       │  2:12            │
  │ STAR     │  "*"       │  2:14            │
  │ ID       │  "n"       │  2:16            │
  │ SEMI     │  ";"       │  2:17            │
  │ RBRACE   │  "}"       │  3:1             │
  └──────────┴────────────┴──────────────────┘
```

**词法分析的关键点：**

- **最长匹配原则**：`>=` 应该识别为一个 `GE` Token，而不是 `>` 和 `=`
- **关键字与标识符的区分**：`int` 是关键字，`integer` 是标识符——需要在识别后再分类
- **空白和注释的处理**：通常被丢弃，但换行符可能需要保留（用于错误定位）
- **错误处理**：遇到非法字符（如 `@`）时需要报告错误

### 1.4.3 阶段二：语法分析（Syntax Analysis）

**任务**：将 Token 流组织为抽象语法树（AST），验证程序是否符合语法规则。

语法分析器（Parser）根据语言的**上下文无关文法（CFG）**，将线性的 Token 流转换为树形结构。

```
以 "return n * n;" 为例，语法分析产生：

       ┌──────────────┐
       │ ReturnStmt   │
       └──────┬───────┘
              │
       ┌──────┴───────┐
       │  BinExpr(*)  │
       └──────┬───────┘
         ┌────┴────┐
         │         │
    ┌────┴──┐  ┌───┴───┐
    │ n     │  │ n     │
    │(ID)   │  │(ID)   │
    └───────┘  └───────┘
```

**完整的 `square` 函数的 AST（简化表示）：**

```
  FunctionDecl("square", return_type=int)
  ├── Parameters:
  │   └── Param("n", type=int)
  └── Body:
      └── ReturnStmt
          └── BinExpr(*)
              ├── ID("n")
              └── ID("n")
```

**语法分析的核心概念：**

- **文法（Grammar）**：用产生式规则描述语言的语法结构
  ```
  statement → return expr ;
  expr      → expr * expr
            | expr + expr
            | ID
            | INT_LIT
  ```
- **自顶向下 vs 自底向上**：递归下降（自顶向下）vs LR 分析（自底向上）
- **歧义处理**：`a + b * c` 应该被解析为 `a + (b * c)`，通过运算符优先级解决

### 1.4.4 阶段三：语义分析（Semantic Analysis）

**任务**：验证程序的语义正确性，收集类型信息，建立符号表。

语法分析只检查程序的**结构**是否正确，语义分析则检查程序的**含义**是否正确。

```
语义分析需要完成的工作：

1. 类型检查
   ┌─────────────────────────────────────┐
   │  int x = "hello";  ← 类型错误！     │
   │  square("hello")   ← 参数类型不匹配  │
   └─────────────────────────────────────┘

2. 作用域解析
   ┌─────────────────────────────────────┐
   │  int square(int n) {                │
   │      return n * n;  ← n 绑定到参数  │
   │  }                                  │
   │  int main() {                       │
   │      n = 5;  ← 错误！n 未在 main    │
   │  }             中定义               │
   └─────────────────────────────────────┘

3. 符号表构建
   ┌──────────────────────────────────────────────┐
   │  Scope: Global                               │
   │  ├── square: Function(int) -> int            │
   │  │   └── Scope: square                       │
   │  │       └── n: int (parameter)              │
   │  ├── sum_of_squares: Function(int) -> int    │
   │  │   └── Scope: sum_of_squares               │
   │  │       ├── n: int (parameter)              │
   │  │       ├── total: int (local)              │
   │  │       └── i: int (local)                  │
   │  └── main: Function() -> int                 │
   │      └── Scope: main                         │
   │          └── result: int (local)             │
   └──────────────────────────────────────────────┘

4. 类型推导（对支持类型推导的语言）
   ┌─────────────────────────────────────┐
   │  var x = 5;          → x: int      │
   │  var y = x + 3.14;   → y: double   │
   └─────────────────────────────────────┘
```

### 1.4.5 阶段四：中间代码生成（IR Generation）

**任务**：将 AST 转换为一种独立于源语言和目标平台的中间表示（Intermediate Representation）。

IR 是编译器的"工作语言"——优化阶段在 IR 上工作，代码生成阶段从 IR 翻译为目标代码。

**常见的 IR 形式：三地址码（Three-Address Code）**

```
# example.c 的三地址码表示

# 函数: square(int n)
square:
    t1 = n * n
    return t1

# 函数: sum_of_squares(int n)
sum_of_squares:
    total = 0
    i = 1
L1:
    if i > n goto L2
    t2 = call square, i
    total = total + t2
    i = i + 1
    goto L1
L2:
    return total

# 函数: main()
main:
    t3 = call sum_of_squares, 5
    result = t3
    return result
```

**三地址码的特点：**
- 每条指令最多包含三个操作数（两个源操作数，一个目标操作数）
- 便于后续的优化分析和变换
- 接近汇编但不依赖具体机器架构

### 1.4.6 阶段五：优化（Optimization）

**任务**：对 IR 进行等价变换，使生成的代码更高效。

优化是编译器中最复杂、最能体现技术水平的阶段。

```
示例：对 sum_of_squares(5) 进行优化

优化前（三地址码）：
    total = 0
    i = 1
L1:
    if i > n goto L2        ← 循环条件
    t2 = call square, i     ← 函数调用开销
    total = total + t2      ← 累加
    i = i + 1
    goto L1
L2:
    return total

─── 常量传播 + 内联 ───

如果编译器知道 n=5（通过常量传播），
并且内联 square 函数：

    total = 0
    i = 1
L1:
    if i > 5 goto L2
    t2 = i * i              ← 内联展开，消除函数调用
    total = total + t2
    i = i + 1
    goto L1
L2:
    return total

─── 循环展开 + 常量折叠 ───

    total = 0
    total = total + 1*1     = 1
    total = total + 2*2     = 5
    total = total + 3*3     = 14
    total = total + 4*4     = 30
    total = total + 5*5     = 55
    return 55               ← 直接返回常量！

─── 最终优化结果 ───

main:
    return 55               ← 编译器直接计算出结果！
```

**常见的优化技术分类：**

| 类别 | 优化技术 | 说明 |
|------|---------|------|
| **局部优化** | 常量折叠、公共子表达式消除 | 在基本块内进行 |
| **循环优化** | 循环不变量外提、循环展开、强度削减 | 针对循环结构 |
| **全局优化** | 死代码消除、常量传播、全局值编号 | 跨基本块 |
| **过程间优化** | 函数内联、过程间常量传播 | 跨函数 |
| **机器相关优化** | 指令调度、寄存器分配、窥孔优化 | 针对目标机器特性 |

### 1.4.7 阶段六：目标代码生成（Code Generation）

**任务**：将优化后的 IR 翻译为目标平台的机器码或汇编代码。

```
# x86-64 汇编（AT&T 语法，GCC 生成的简化版本）

# 函数: square(int n) — n 在 %edi 寄存器中
square:
    movl    %edi, %eax      # eax = n
    imull   %edi, %eax      # eax = n * n
    ret                     # 返回 eax

# 函数: sum_of_squares(int n) — n 在 %edi 寄存器中
sum_of_squares:
    pushq   %rbp            # 保存帧指针
    movq    %rsp, %rbp      # 建立栈帧
    pushq   %rbx            # 保存 callee-saved 寄存器
    subq    $8, %rsp        # 分配局部变量空间
    movl    %edi, %ebx      # ebx = n
    movl    $0, -12(%rbp)   # total = 0
    movl    $1, -8(%rbp)    # i = 1
.L2:
    cmpl    %ebx, -8(%rbp)  # 比较 i 和 n
    jg      .L3             # if i > n, 跳出循环
    movl    -8(%rbp), %edi  # 参数 = i
    call    square           # 调用 square(i)
    addl    %eax, -12(%rbp) # total += 返回值
    addl    $1, -8(%rbp)    # i++
    jmp     .L2             # 继续循环
.L3:
    movl    -12(%rbp), %eax # 返回 total
    addq    $8, %rsp        # 恢复栈
    popq    %rbx            # 恢复寄存器
    popq    %rbp            # 恢复帧指针
    ret

# 函数: main()
main:
    pushq   %rbp
    movq    %rsp, %rbp
    movl    $5, %edi        # 参数 = 5
    call    sum_of_squares   # 调用 sum_of_squares(5)
    # 返回值已在 eax 中
    popq    %rbp
    ret
```

**代码生成阶段的关键任务：**

| 任务 | 说明 |
|------|------|
| **指令选择** | 将 IR 操作映射为目标机器的指令 |
| **寄存器分配** | 决定哪些变量放在寄存器中（NP-hard 问题，常用图着色算法） |
| **指令调度** | 重排指令顺序以利用流水线（避免数据冒险） |
| **栈帧布局** | 分配局部变量、参数传递的空间 |

## 1.5 关键数据结构概览

编译器的每个阶段都会产生或使用特定的数据结构。理解这些数据结构是理解编译器工作原理的关键。

### 1.5.1 Token（词法单元）

Token 是词法分析的输出，表示源代码中的一个基本语法单位。

```c
// Token 的典型数据结构
typedef enum {
    // 关键字
    TOKEN_INT, TOKEN_RETURN, TOKEN_FOR, TOKEN_IF, TOKEN_WHILE,
    // 标识符和字面量
    TOKEN_IDENTIFIER, TOKEN_INTEGER_LIT,
    // 运算符
    TOKEN_PLUS, TOKEN_MINUS, TOKEN_STAR, TOKEN_SLASH,
    TOKEN_ASSIGN, TOKEN_EQ, TOKEN_NEQ,
    TOKEN_LT, TOKEN_GT, TOKEN_LE, TOKEN_GE,
    // 分隔符
    TOKEN_LPAREN, TOKEN_RPAREN, TOKEN_LBRACE, TOKEN_RBRACE,
    TOKEN_SEMICOLON, TOKEN_COMMA,
    // 特殊
    TOKEN_EOF, TOKEN_ERROR
} TokenKind;

typedef struct {
    TokenKind kind;      // Token 类型
    const char *lexeme;  // 原始文本
    int line;            // 行号
    int column;          // 列号
    union {
        int int_value;   // 整数字面量的值
        // 可扩展：float_value, string_value 等
    };
} Token;
```

**Token 的作用**：
- 是编译器前端最基本的输出单元
- 为后续阶段提供位置信息（用于错误报告）
- 分离了字符级别的处理和语法级别的处理

### 1.5.2 AST（抽象语法树）

AST 是语法分析的输出，以树形结构表示程序的语法结构。"抽象"意味着它省略了语法中的冗余信息（如括号、分号等），只保留结构本质。

```c
// AST 节点的典型定义（简化版）

// 表达式节点
typedef enum {
    EXPR_INT_LIT,       // 整数字面量: 42
    EXPR_IDENTIFIER,    // 标识符: x
    EXPR_BINARY,        // 二元运算: a + b
    EXPR_UNARY,         // 一元运算: -x
    EXPR_CALL,          // 函数调用: f(x, y)
} ExprKind;

typedef struct Expr {
    ExprKind kind;
    Type *type;         // 语义分析后填充的类型信息
    union {
        int int_literal;
        const char *identifier;
        struct { struct Expr *left, *right; TokenKind op; } binary;
        struct { struct Expr *operand; TokenKind op; } unary;
        struct { const char *func_name; struct Expr **args; int arg_count; } call;
    };
} Expr;

// 语句节点
typedef enum {
    STMT_RETURN,        // return expr;
    STMT_ASSIGN,        // x = expr;
    STMT_FOR,           // for(init; cond; update) body
    STMT_IF,            // if(cond) then else
    STMT_BLOCK,         // { stmt1; stmt2; ... }
    STMT_EXPR,          // expr; (表达式语句)
    STMT_DECL,          // int x = expr;
} StmtKind;

// 声明/定义节点
typedef enum {
    DECL_FUNCTION,      // 函数定义
    DECL_VARIABLE,      // 变量声明
} DeclKind;
```

**example.c 中 `square` 函数的 AST 结构（可视化）：**

```
Decl_Function
├── name: "square"
├── return_type: Type_Int
├── params:
│   └── Param { name: "n", type: Type_Int }
└── body:
    └── Stmt_Block
        └── Stmt_Return
            └── Expr_Binary (*)
                ├── left: Expr_Identifier("n")
                └── right: Expr_Identifier("n")
```

### 1.5.3 符号表（Symbol Table）

符号表记录了程序中所有标识符（变量、函数、类型等）的信息，是语义分析的核心数据结构。

```c
// 符号表的典型实现（使用哈希表 + 作用域链）

typedef struct Symbol {
    const char *name;       // 标识符名称
    Type *type;             // 类型信息
    enum {
        SYMBOL_VARIABLE,
        SYMBOL_FUNCTION,
        SYMBOL_PARAMETER,
        SYMBOL_TYPE,
    } kind;
    // 额外信息
    int scope_level;        // 作用域层级
    int offset;             // 在栈帧中的偏移（代码生成时使用）
    bool is_defined;        // 是否已定义（不仅仅是声明）
    struct Symbol *next;    // 哈希表链表
} Symbol;

typedef struct Scope {
    Symbol *table[HASH_SIZE];  // 哈希表桶数组
    struct Scope *parent;      // 父作用域（作用域链）
    const char *name;          // 作用域名称（函数名等）
} Scope;

// example.c 的符号表状态：

// 全局作用域
// ┌─────────────────────────────────────────┐
// │ Scope: global                            │
// │ ┌──────────────────┬───────────────────┐ │
// │ │ Name             │ Info              │ │
// │ ├──────────────────┼───────────────────┤ │
// │ │ square           │ Function(int)->int│ │
// │ │ sum_of_squares   │ Function(int)->int│ │
// │ │ main             │ Function()->int   │ │
// │ └──────────────────┴───────────────────┘ │
// └─────────────────────────────────────────┘
//
// square 的局部作用域
// ┌─────────────────────────────────────────┐
// │ Scope: square (parent: global)           │
// │ ┌──────────────────┬───────────────────┐ │
// │ │ Name             │ Info              │ │
// │ ├──────────────────┼───────────────────┤ │
// │ │ n                │ int (parameter)   │ │
// │ └──────────────────┴───────────────────┘ │
// └─────────────────────────────────────────┘
//
// sum_of_squares 的局部作用域
// ┌─────────────────────────────────────────┐
// │ Scope: sum_of_squares (parent: global)   │
// │ ┌──────────────────┬───────────────────┐ │
// │ │ Name             │ Info              │ │
// │ ├──────────────────┼───────────────────┤ │
// │ │ n                │ int (parameter)   │ │
// │ │ total            │ int (local, off=0)│ │
// │ │ i                │ int (local, off=4)│ │
// │ └──────────────────┴───────────────────┘ │
// └─────────────────────────────────────────┘
```

### 1.5.4 IR（中间表示）

IR 是介于源语言和目标语言之间的中间代码表示。它是优化的主要工作对象。

**IR 的主要形式：**

| 形式 | 说明 | 代表 |
|------|------|------|
| **三地址码** | 每条指令最多三个操作数 | GCC 的 GIMPLE（优化前） |
| **SSA（静态单赋值）** | 每个变量只被赋值一次 | LLVM IR、GCC 的 RTL |
| **控制流图（CFG）** | 以基本块为节点的有向图 | 所有现代编译器 |

**LLVM IR 示例（example.c 的 square 函数）：**

```llvm
; LLVM IR 文本格式
define i32 @square(i32 %n) {
entry:
  %mul = mul i32 %n, %n
  ret i32 %mul
}

define i32 @sum_of_squares(i32 %n) {
entry:
  br label %loop

loop:
  %i = phi i32 [ 1, %entry ], [ %i.next, %loop.body ]
  %total = phi i32 [ 0, %entry ], [ %total.next, %loop.body ]
  %cmp = icmp sgt i32 %i, %n
  br i1 %cmp, label %exit, %loop.body

loop.body:
  %sq = call i32 @square(i32 %i)
  %total.next = add i32 %total, %sq
  %i.next = add i32 %i, 1
  br label %loop

exit:
  ret i32 %total
}
```

> 注意 LLVM IR 中的 `phi` 指令——它是 SSA 形式的核心特征，用于在控制流汇合点选择变量的值。

### 1.5.5 CFG（控制流图）

CFG 用图的形式表示程序的执行路径，是优化分析的基础数据结构。

```
sum_of_squares 函数的控制流图：

    ┌───────────────┐
    │   Entry Block  │
    │   total = 0    │
    │   i = 1        │
    └───────┬───────┘
            │
            ▼
    ┌───────────────┐
    │  Loop Header   │ ← 循环头（循环条件）
    │  if i > n      │
    └───┬───────┬───┘
        │       │
   true │       │ false
        ▼       ▼
    ┌───────┐ ┌───────────────┐
    │  Exit │ │  Loop Body     │
    │       │ │  t = square(i) │
    │ return│ │  total += t    │
    │ total │ │  i++           │
    └───────┘ └───────┬───────┘
                       │
                       │ (跳回 Loop Header)
                       │
                       └───→ Loop Header

    节点 = 基本块（Basic Block）
    边 = 控制流转移（跳转/分支）

    基本块的定义：
    - 只有一个入口（第一条指令）
    - 只有一个出口（最后一条指令）
    - 内部没有跳转目标，也没有分支
```

**CFG 上的典型分析：**

| 分析类型 | 用途 |
|---------|------|
| **活跃变量分析** | 哪些变量在某点之后还会被使用？（用于寄存器分配） |
| **到达定义分析** | 某个变量的赋值可能到达哪些使用点？（用于常量传播） |
| **支配关系分析** | 哪个基本块支配（dominate）另一个？（用于 SSA 构建） |
| **循环识别** | 识别自然循环及其嵌套结构（用于循环优化） |

## 本章小结

本章建立了编译器和解释器的全局认知框架：

1. **编译器**将源代码翻译为目标代码，经历预处理→编译→汇编→链接四个阶段
2. **解释器**直接执行源代码或中间表示，CPython 采用字节码虚拟机，V8 和 LuaJIT 采用分层 JIT 策略
3. **现代语言**普遍采用混合模式，结合编译和解释的优势
4. **编译器的核心阶段**：词法分析 → 语法分析 → 语义分析 → IR 生成 → 优化 → 代码生成
5. **关键数据结构**：Token、AST、符号表、IR、CFG 贯穿整个编译过程
6. **example.c** 将作为贯穿整个系列的示例，展示每个阶段如何处理同一段代码

下一章（第 2 章：编译器架构）将深入探讨编译器的内部组织方式，包括模块化设计、错误处理策略和编译驱动程序的实现。
